# rosay4.github.io

这是我的个人网站仓库，目前采用多页面静态结构，整体分为两个主要部分：

- `Home`：主页，承担个人入口与概览信息
- `About / CV`：个人简介、背景、链接与简历信息
- `Archive`：博客归档页
- `Archive/<slug>/`：未来单篇文章页的目录结构

当前阶段的目标不是立刻上线完整内容，而是先把网站框架、信息架构和整体视觉语言搭好。

## 设计方向

本站当前的视觉方向以 `Thinking Machines` 的文章页风格为主要参考：

- 白底、低饱和、克制排版
- 强标题层级
- 细分隔线
- 更接近研究出版物，而不是常见卡片式博客模板

信息组织上会参考个人主页 / CV 页面，但视觉主导不是传统作品集风格，而是更偏“研究、写作、笔记”的站点气质。

- `archive/<slug>/`
  - 未来单篇文章页路径约定

## 后续是接 Hexo 还是继续自建

当前阶段更建议继续自建，而不是马上切到 Hexo。

原因如下：

- 现在最重要的是把网站的视觉和信息架构做对
- 你想要的是比较强的定制化 UI，而不是快速套一个博客主题
- Hexo 很适合“我要尽快开始发文章”，但不一定适合“我要先把站点气质做准”

更合适的路线是：

1. 先保留当前这套手写静态框架
2. 等页面结构稳定后，再决定是否接入文章系统
3. 如果未来文章数量明显变多，再评估是否：
   - 接入 Hexo
   - 或继续使用自定义静态结构加简单脚本生成

如果未来真的引入 Hexo，比较推荐的方式不是推翻当前页面重做，而是：

- 把当前视觉样式保留下来
- 将 Hexo 只当作文章内容和归档的生成层
- 按现在的设计语言去定制主题

也就是说，Hexo 更适合作为“内容生成工具”，而不是“先决定整个网站长什么样”。

## 维护约定

- 主页使用多页面结构，不是单页滚动站
- 公共样式集中在 `styles.css`
- 单篇文章路径约定为 `/archive/<slug>/`
- 内容优先保持简洁，避免为了未来可能性过度抽象
